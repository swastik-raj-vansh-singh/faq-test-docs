# Logger Node

<SectionRows section="Node Type" chips={["Data"]} />

The Logger Node writes a structured log entry at any point in a flow without interrupting execution. It captures upstream data, attaches optional metadata, and routes the entry to one or more configured destinations — the Lamatic run log, an external logging service, or a database table. The original data passes through unchanged so the Logger Node can be inserted anywhere in a pipeline without altering its behavior.

## How It Works

1. The Logger Node receives the current flow context from an upstream node.
2. It assembles a log entry from the configured fields and any static metadata you provide.
3. The entry is written to the configured destination(s) asynchronously.
4. The original data is forwarded to the next node exactly as received.

## Configuration

| Setting | Description | Required |
|---|---|---|
| **Label** | Human-readable name for this log point, shown in the run log UI | Yes |
| **Fields** | Template references to the values to capture, e.g. `{{llmNode_1.output.text}}` | Yes |
| **Level** | Log severity: `debug`, `info`, `warn`, or `error` | No (defaults to `info`) |
| **Destination** | Where to write the entry: `run_log`, `external` (HTTP endpoint), or `database` | No (defaults to `run_log`) |
| **Metadata** | Static key-value pairs appended to every entry, such as environment or version tags | No |
| **Mask fields** | List of field names whose values are redacted before logging | No |

### Destinations

| Destination | Description |
|---|---|
| `run_log` | Entries appear in the Lamatic flow run detail view alongside node outputs |
| `external` | Entries are POSTed as JSON to a configured HTTP endpoint (e.g. Datadog, Logtail, Axiom) |
| `database` | Entries are inserted as rows into a connected database integration |

## When to Use It

- **Debugging**: Inspect intermediate values between two nodes without modifying the pipeline structure.
- **Audit trails**: Record every decision made by a Conditional or Router Node for compliance purposes.
- **LLM observability**: Log prompts, completions, token counts, and latency at each LLM call.
- **Error capture**: Place a Logger Node on the error branch of a Validator Node to persist failure details before the flow exits.

<Callout type="info">
Logger Node writes are asynchronous — they do not add latency to the critical path of your flow. However, entries may appear in external destinations with a short delay (typically under one second).
</Callout>

## Output Fields

The Logger Node forwards all upstream outputs unchanged and adds:

- `{{loggerNode_1.output.entryId}}` — unique identifier of the log entry that was written
- `{{loggerNode_1.output.logged}}` — `true` if the entry was successfully dispatched to the destination

## Masking Sensitive Data

Use the **Mask fields** setting to redact values before they reach any destination. Masked fields are replaced with `[REDACTED]` in the stored entry. This is useful for PII such as email addresses, API keys passed through a flow, or any value that should not appear in logs.

<Callout type="warning">
Masking applies only to the Logger Node's own output destination. It does not redact values from other nodes or from the main run log entries written by the Lamatic platform itself.
</Callout>

## Example: Log LLM Inputs and Outputs for Observability

<Steps>
### Step 1
A Webhook Trigger receives a user question and passes it to an LLM Node.

### Step 2
A Logger Node is placed immediately after the LLM Node with:
- **Label**: `LLM call`
- **Fields**: `{{webhookTrigger.output.body.question}}`, `{{llmNode_1.output.text}}`, `{{llmNode_1.output.usage.totalTokens}}`
- **Level**: `info`
- **Destination**: `external` pointing to your Axiom dataset endpoint

### Step 3
Every time the flow runs, the question, the model response, and the token count are captured in Axiom without adding any measurable latency to the user-facing response.

### Step 4
The LLM output continues to the Response Node and is returned to the caller as normal.
</Steps>

## Performance Considerations

- Log writes to `run_log` are in-memory and add negligible overhead.
- Writes to `external` destinations involve an outbound HTTP request made asynchronously off the critical path; network latency does not affect flow execution time.
- Writes to `database` destinations use the same connection pool as other database nodes and count against your connection limit.
- Avoid logging very large payloads (multi-megabyte strings or arrays) to external destinations on high-throughput flows; prefer logging summary fields or counts instead.

## Limitations

- A single Logger Node can capture up to 50 fields per entry.
- Log entries written to `run_log` are retained according to your project's run history retention policy.
- The Logger Node cannot modify or filter data — use a Data Mapper Node before logging if you need to reshape the entry.
- Circular references in the captured data (objects that reference themselves) will cause the serialization step to fail; flatten such structures upstream before logging.

## Related

See **Conditional Node** and **Router Node** for branching logic whose decisions you may want to audit, **Validator Node** for generating error-branch entries worth capturing, and **Data Mapper** for reshaping data into a cleaner structure before it is logged.
