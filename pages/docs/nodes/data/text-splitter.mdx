# Text Splitter Node

<SectionRows section="Node Type" chips={["Data"]} />

The Text Splitter Node breaks large documents into smaller, overlapping chunks before they are passed to the Vectorize and Index nodes. Chunking is a critical step in any RAG pipeline — embedding models have token limits, and retrieving smaller, focused chunks improves the relevance of search results.

## How It Works

When a document is too long to be embedded as a whole, it must be split into chunks that:

- Fit within the embedding model's token limit
- Are small enough to be precise when retrieved
- Overlap slightly so no context is lost at chunk boundaries

The Text Splitter Node splits text using a configurable strategy and passes each chunk downstream as a separate item.

## Configuration

| Parameter | Description | Required |
|---|---|---|
| **Text** | The input text or document to split | Yes |
| **Chunk Size** | Maximum number of tokens or characters per chunk | Yes |
| **Chunk Overlap** | Number of tokens or characters to overlap between chunks | No |
| **Split Strategy** | How to split: `recursive`, `sentence`, `paragraph`, or `fixed` | No |
| **Separator** | Custom separator string used for splitting (for fixed strategy) | No |

## Setting Up Text Splitter

<Steps>

### Step 1: Add the Text Splitter Node

Drag the Text Splitter Node from the Data section of the node palette into your flow. Place it after your document source (e.g., File Loader Node) and before the Vectorize Node.

### Step 2: Map the Text Input

Set the text field to the variable containing the raw document content:

```
{{fileLoaderNode_1.output.text}}
```

### Step 3: Set Chunk Size

Choose a chunk size appropriate for your embedding model. Most models support up to 512 or 8192 tokens. A chunk size of 300–500 tokens typically gives good retrieval precision.

<Callout type="warning">
Do not set the chunk size larger than the token limit of your embedding model. Oversized chunks will be truncated, causing silent data loss during vectorization.
</Callout>

### Step 4: Set Chunk Overlap

Set an overlap of 10–20% of your chunk size. For example, if chunk size is 400 tokens, an overlap of 50–80 tokens preserves context at chunk boundaries.

### Step 5: Choose a Split Strategy

Select the strategy that best fits your content type:

- **Recursive**: Tries to split on paragraphs, then sentences, then words. Best for general-purpose text.
- **Sentence**: Splits on sentence boundaries. Good for narrative or conversational content.
- **Paragraph**: Splits on paragraph breaks. Good for structured documents.
- **Fixed**: Splits at exact character or token counts. Use when structure doesn't matter.

</Steps>

## Output Schema

The Text Splitter Node outputs an array of chunk objects:

| Field | Type | Description |
|---|---|---|
| `text` | string | The text content of the chunk |
| `index` | number | The position of the chunk in the original document |
| `metadata` | object | Metadata passed through from the input, plus chunk index |

## Split Strategies Compared

| Strategy | Best For | Preserves Structure |
|---|---|---|
| Recursive | General documents, mixed content | Yes |
| Sentence | Conversational or narrative text | Partially |
| Paragraph | Reports, articles, documentation | Yes |
| Fixed | Logs, code, structured data | No |

## Best Practices

- **Use recursive splitting by default**: It handles most content types gracefully and avoids cutting mid-sentence.
- **Keep chunks focused**: Smaller chunks (200–400 tokens) tend to produce more precise retrieval results than large ones.
- **Always use overlap**: A 10–15% overlap ensures that sentences or facts split at chunk boundaries are still retrievable from either chunk.
- **Pass metadata through**: Include source file name, page number, or section title in the metadata so downstream nodes can filter or display context alongside results.
- **Test chunk quality**: Use the test panel to inspect how your document is being split before indexing. Poor splits are a common source of bad RAG results.

## Common Issues

### Chunks are cut mid-sentence

**Cause**: Using the fixed strategy with a character count that does not align with sentence boundaries.
**Fix**: Switch to the recursive or sentence strategy, which respects natural language boundaries.

### Too many small chunks being created

**Cause**: Chunk size is set too low for the content.
**Fix**: Increase chunk size. For most prose content, 300–500 tokens is a good starting range.

### Context is lost between retrieved chunks

**Cause**: Chunk overlap is set to zero or too low.
**Fix**: Increase chunk overlap to 10–20% of the chunk size to ensure boundary context is preserved.

## Related Nodes

- **File Loader Node**: Loads files and extracts raw text, which is then passed to this node for splitting.
- **Vectorize Node**: Converts text chunks into embeddings. Always place after the Text Splitter Node.
- **Index Node**: Stores vectorized chunks in your vector database.
- **Vector Search Node**: Retrieves the most relevant chunks at query time.
