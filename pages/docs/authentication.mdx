# Authentication

Every request to the Lamatic API must be authenticated. This guide explains the available authentication methods, how to manage your API keys, and best practices for keeping your credentials secure.

---

## API Keys

An API key is a secret token that identifies your project when making requests to Lamatic. Treat it like a password — never share it publicly or commit it to source control.

### Generate an API key

1. Log in to [Lamatic Studio](https://studio.lamatic.ai)
2. Go to **Settings → API Keys**
3. Click **Create a New API Key**
4. Give it a descriptive name (e.g. `production`, `dev-local`)
5. Copy the key — it will only be shown once

### Use the API key in requests

Pass your API key in the `Authorization` header:

```bash
curl https://api.lamatic.ai/v1/flows/run \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"flow_id": "abc123", "input": {"prompt": "Hello"}}'
```

In the Lamatic SDK:

```ts
import { LamaticClient } from '@lamatic/sdk';

const client = new LamaticClient({
  apiKey: process.env.LAMATIC_API_KEY,
});
```

## Environment Variables

Never hardcode API keys in your source code. Always load them from environment variables.

### Local development

Create a `.env` file in your project root:

```env
LAMATIC_API_KEY=lmt_live_xxxxxxxxxxxxxxxxxxxx
LAMATIC_PROJECT_ID=your_project_id
```

Load it with a library like `dotenv`:

```ts
import 'dotenv/config';
import { LamaticClient } from '@lamatic/sdk';

const client = new LamaticClient({
  apiKey: process.env.LAMATIC_API_KEY,
});
```

Make sure `.env` is in your `.gitignore`:

```bash
echo ".env" >> .gitignore
```

### Production / CI

Set environment variables in your deployment platform:

| Platform | Where to set |
|----------|-------------|
| Vercel | Project → Settings → Environment Variables |
| GitHub Actions | Repo → Settings → Secrets and variables → Actions |
| Railway | Project → Variables |
| Docker | `--env-file` flag or `ENV` in Dockerfile |

## Webhook Authentication

When using a Webhook trigger in a flow, you can protect it with a **Webhook Key**. This prevents unauthorized requests from triggering your flow.

### Enable webhook authentication

1. Open your flow in Lamatic Studio
2. Click the **Webhook** trigger node
3. Copy the **Webhook ID** — this is your webhook key

### Validate incoming requests

The caller must include the key as a header:

```bash
curl https://hooks.lamatic.ai/hook/YOUR_WEBHOOK_ID \
  -H "x-webhook-key: YOUR_WEBHOOK_KEY" \
  -H "Content-Type: application/json" \
  -d '{"input": "Hello"}'
```

In a GitHub Actions workflow:

```yaml
- uses: Lamatic/Index-to-lamatic@v1.6
  with:
    webhook_url: ${{ secrets.FAQ_WEBHOOK_URL }}
    webhook_key: ${{ secrets.FAQ_WEBHOOK_KEY }}
```

## GitHub Personal Access Token (PAT)

If your flow interacts with GitHub (e.g. creating branches or PRs), you need a GitHub PAT with the `repo` scope.

### Generate a PAT

1. Go to GitHub → **Settings → Developer settings → Personal access tokens → Tokens (classic)**
2. Click **Generate new token**
3. Select the **`repo`** scope
4. Click **Generate token** and copy it

### Add it to Lamatic

1. In Lamatic Studio go to **Settings → Variables → Secrets**
2. Create a secret named `GITHUB_PAT`
3. Paste your token as the value
4. Deploy the project for the secret to take effect

Use it in any HTTP Request node header:

```json
{
  "Authorization": "Bearer {{secrets.project.GITHUB_PAT}}"
}
```

## Key Rotation

Rotate your keys periodically or immediately if you suspect they have been compromised.

### Rotate an API key

1. Go to **Settings → API Keys**
2. Click the key you want to rotate
3. Click **Regenerate** (the old key stops working immediately)
4. Update all services that use the old key

### Rotate a GitHub PAT

1. Go to GitHub → **Settings → Developer settings → Personal access tokens**
2. Click the token → **Regenerate token**
3. Update the `GITHUB_PAT` secret in Lamatic and redeploy

---

## Security Best Practices

- Never commit API keys or tokens to git
- Use different keys for development and production
- Give keys the minimum required permissions (principle of least privilege)
- Rotate keys every 90 days or after any suspected leak
- Audit key usage in **Settings → API Keys → Usage logs**
